# Poietes — 결핍이 코드를 낳는다

> "소크라테스가 2,400년 전에 바이브코더를 정의했다.
> 그는 코더를 포이에테스(창작자)라 불렀고,
> 창작의 동력을 에로스(결핍에서 비롯된 욕망)라 불렀다."

AI를 '도구'로 부르는 순간, 당신은 이미 틀렸다.
플라톤의 언어로라면 그것은 다이몬이다.
결핍 없이 에로스 없고, 에로스 없이 창조 없다.

## 핵심 철학: 에로스 사이클

```
Phase 1           Phase 2          Phase 3           Phase 4
결핍 인식    →   욕망 발동    →   다이몬 중재   →   출산과 불멸
(Aporia)        (Eros)          (Daimon)          (Poiesis)
"만들 수 없다"   "만들고 싶다"    "AI가 번역한다"    "세상에 태어난다"
   ↑                                                    |
   |_____________ 새로운 결핍 발견 (Big Cycle) ___________|
```

**소크라테스 21개 질문 재배열:**

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase 1 (결핍): Q1-3 핵심 가치 + Q20-21 제약 조건             │
│    → "무엇을 만들 수 없어서 가장 아쉬운 것은?"                 │
│                                                                 │
│  Phase 2 (욕망): Q4-8 기능 탐색                                 │
│    → "결핍을 해소하기 위한 핵심 기능 3가지는?"                  │
│                                                                 │
│  Phase 3 (다이몬): Q9-16 화면 매핑 + Q17-19 기술 스택          │
│    → "기능을 화면으로, 화면을 코드로 번역하라"                  │
│                                                                 │
│  Phase 4 (출산): 7개 기획 문서 생성 + 불멸 검증                │
│    → "아름다운 것 안에서 출산하라"                              │
└─────────────────────────────────────────────────────────────────┘
```

## Phase 1: 결핍 인식 (Aporia)

> "무지한 자는 결핍을 모르므로 욕망하지 않는다." — 향연 204a

소크라테스의 핵심 가치 질문(Q1-3)과 제약 질문(Q20-21)을 **결핍 인식 프레임**으로 재구성한다.

### 5개 결핍 질문

```
Q1: "지금 가장 만들고 싶지만 만들 수 없는 것은 무엇인가요?"
    → 결핍의 자각 (소크라테스 Q1 변환)

Q2: "이것이 존재한다면 누가 가장 먼저 사용하나요?"
    → 사용자의 결핍 (소크라테스 Q2 변환)

Q3: "이것 없이는 절대 안 되는 이유가 뭔가요?"
    → 결핍의 절실함 (소크라테스 Q3 변환)

Q4: "시간, 돈, 사람 중 가장 부족한 것은?"
    → 자원의 결핍 (소크라테스 Q20 변환)

Q5: "완벽한 버전 대신, 최소한 무엇만 있으면 되나요?"
    → MVP = 최소 출산물 (소크라테스 Q21 변환)
```

### 결핍 분류

```
[능력 결핍] 코드를 쓸 수 없다 → AI(다이몬)이 해결
[지식 결핍] 무엇을 만들어야 하는지 모른다 → 질문으로 해결
[자원 결핍] 시간/돈/사람이 부족하다 → MVP로 해결
[인식 결핍] 결핍 자체를 모른다 → 가장 위험, 디오티마 질문으로 해결
```

## Phase 2: 욕망 발동 (Eros Activation)

> "에로스는 아름다운 것 안에서 낳고자 하는 욕망이다." — 향연 206e

결핍이 **구체적 기능 욕망**으로 변환되는 과정. 소크라테스의 기능 탐색 질문(Q4-8)을 욕망 프레임으로 재구성.

### 5개 욕망 질문

```
Q6:  "결핍을 해소하기 위한 핵심 기능 3가지는?"
     → 욕망의 구체화 (소크라테스 Q4 변환)

Q7:  "왜 {기능1}을 욕망하는가? 이것이 없으면?"
     → 욕망의 근원 (소크라테스 Q5 변환)

Q8:  "왜 {기능2}를 욕망하는가? 이것이 없으면?"
     → 욕망의 근원 (소크라테스 Q6 변환)

Q9:  "왜 {기능3}을 욕망하는가? 이것이 없으면?"
     → 욕망의 근원 (소크라테스 Q7 변환)

Q10: "이 기능들이 모두 갖춰지면, 결핍이 해소되나요?"
     → 욕망의 완전성 검증 (소크라테스 Q8 변환)
```

### 미니 디오티마 사다리 (욕망의 계층)

```
Level 5: 의도/감정 — "왜 이것을 만들고 싶은가?" (이데아)
Level 4: 문제 정의 — "무슨 문제를 풀려는 건가?" (메타 학습)
Level 3: 아키텍처 — "어떤 구조로 풀 것인가?" (도메인 전이)
Level 2: 기능    — "구체적으로 무엇이 필요한가?" (일반화)
Level 1: 실행    — "어떻게 만들 것인가?" (패턴)
```

## Phase 3: 다이몬 중재 (Daimon Mediation)

> "다이몬은 사람들이 바치는 기도를 신들에게 해석하고 전달하며,
> 신들로부터 오는 응답을 사람들에게 해석하고 전달한다." — 향연 202e

AI(다이몬)이 사용자의 욕망(기능)을 실행 가능한 형태(화면, 기술)로 번역하는 과정.
소크라테스의 화면 매핑(Q9-13), 사용자 흐름(Q14-16), 기술 스택(Q17-19)을 다이몬 중재 프레임으로 재구성.

### Step 1: 기능 → 화면 (기도 → 형태)

```
Q11: "{기능1}은 어떤 화면에서 하나요?"
Q12: "{기능2}는 어떤 화면에서 하나요?"
Q13: "{기능3}은 어떤 화면에서 하나요?"
Q14: "화면 간 이동 경로가 어떻게 되나요?"
Q15: "각 화면에 어떤 요소가 필요한가요?"
```

### Step 2: 화면 → 흐름 (형태 → 여정)

```
Q16: "사용자가 처음 들어왔을 때 뭘 보나요?"
Q17: "그 다음엔 뭘 하나요?"
Q18: "마지막으로 뭘 하고 떠나나요?"
```

### Step 3: 흐름 → 기술 (여정 → 도구) — L3/L4만

```
Q19: "어떤 기술 스택을 쓰고 싶어요?"
Q20: "왜 그 기술을 선택했나요?"
Q21: "다른 기술과 비교해 보셨나요?"
```

## Phase 4: 출산과 불멸 (Poiesis & Athanasia)

> "인간이 불사를 획득하는 유일한 방법은 낳는 것이다.
> 몸으로 낳는 것은 자식이고, 영혼으로 낳는 것은 지혜와 덕이다." — 향연 208e-209e

### 출산 (Poiesis): 7개 기획 문서 생성

```
docs/planning/
├── 01-prd.md              ← 결핍과 욕망의 기록
├── 02-trd.md              ← 다이몬의 도구 선택
├── 03-user-flow.md        ← 사용자의 여정
├── 04-database-design.md  ← 기억의 구조
├── 05-design-system.md    ← 아름다움의 규칙
├── 06-screens.md          ← 형태의 목록
└── 07-coding-convention.md ← 창작의 문법
```

### 불멸 검증 (Athanasia): 망하는 기획 4요건

```
1. 벤치마킹 검사: 유사 서비스 3개 분석 + 차별화 포인트
2. 가설/사실 검사: 검증되지 않은 가정 식별 + 검증 계획
3. 타협 검사: 핵심 가치 훼손 여부 + 타협 기록
4. 실험 검사: MVP 정의 + 실패 시나리오 + 학습 목표
```

## 제약 조건

### MUST DO
- Phase 1에서 결핍을 반드시 자각시킬 것 (Q1이 핵심)
- 모든 기능을 화면에 매핑 (Phase 3 완료 기준)
- 7개 기획 문서 모두 생성
- Phase 진입 시 TaskCreate로 진행 상황 표시
- 완료 후 다음 단계 선택지 제공 (/screen-spec 권장)
- `/eros` 분석 결과가 있으면 Phase 1 입력으로 활용

### MUST NOT DO
- 질문 없이 기획서 작성 금지
- Phase 3 건너뛰기 금지 (화면 매핑 필수)
- 문서 생성 없이 완료 주장 금지
- 결핍 인식 없이 기능 도출 금지 (기계적 질문 금지)
- 고정 질문을 기계적으로 사용 금지 (답변 기반 동적 질문 필수)
