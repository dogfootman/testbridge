# 결핍 인식 + 욕망 발동 (Phase 1-2)

> **로드 시점**: `/eros` 실행 직후, Phase 1-2 진행 시

## Phase 1: 결핍 인식 (Aporia)

> "지혜로운 자는 이미 지혜롭기에 지혜를 구하지 않는다.
> 무지한 자는 결핍을 모르므로 지혜를 구하지 않는다.
> 오직 결핍을 아는 자만이 욕망한다." — 향연 204a

### 4가지 결핍 유형

결핍 스캔 시 아래 4가지 유형을 모두 점검한다:

| 유형 | 코드 | 설명 | 위험도 | 탐지 방법 |
|------|------|------|--------|----------|
| 명시적 결핍 | `[explicit-gap]` | 에러, 테스트 실패, 빌드 에러 등 이미 드러난 문제 | 낮음 | 에러 로그, 테스트 결과 |
| 숨은 결핍 | `[hidden-gap]` | 모른다는 것조차 모르는 상태 (가장 위험) | 최고 | `/common-ground --check`, 가정 검증 |
| 착각 결핍 | `[assumed-known]` | 안다고 착각하는 것 (던닝-크루거) | 높음 | `/the-fool` Mode 1, 증거 요구 |
| 경계 결핍 | `[boundary-gap]` | 도메인 경계에서 발생하는 결핍 | 중간 | 인터페이스/API 경계 점검 |

### 결핍 스캔 절차

**Step 1: 맥락 수집**

사용자 입력 분석:
- `/eros "주제"` 형식이면 → 주제를 시작점으로 사용
- `/eros` 단독이면 → AskUserQuestion으로 주제 파악
- `/common-ground --check` 결과가 .claude/docs/에 있으면 → 가정 목록을 결핍 후보로 활용
- `/socrates` 기획 문서가 `docs/planning/`에 있으면 → 기획서의 전제/가정을 결핍 후보로 활용
  - 특히 01-prd.md의 핵심 기능, 03-ux.md의 사용자 시나리오에서 `[assumed-known]` 탐지
  - 기획서가 "왜 이 기능이 필요한가?"에 답하지 못하면 → `[hidden-gap]`

**Step 2: AskUserQuestion 3개로 결핍 드러내기**

**Step 3: 결핍 목록 생성**

응답을 분석하여 결핍 목록을 구조화한다:

```markdown
## 결핍 목록

### 명시적 결핍 [explicit-gap]
- [ ] {에러/실패 사례}

### 숨은 결핍 [hidden-gap]
- [ ] {모른다는 것조차 몰랐던 것} ← "확신 영역"의 응답을 의심하여 발견

### 착각 결핍 [assumed-known]
- [ ] {"확실히 안다"고 했지만 증거 없는 것} ← 던닝-크루거 지점

### 경계 결핍 [boundary-gap]
- [ ] {도메인 경계에서 발생하는 결핍}
```

### 던닝-크루거 탐지 규칙

사용자가 "확실히 안다"고 선택한 항목에 대해:

1. 증거를 요청한다 (코드, 문서, 테스트 결과 등)
2. 증거가 없으면 → `[assumed-known]`으로 재분류
3. 증거가 있지만 불완전하면 → `[boundary-gap]`으로 분류
4. 증거가 충분하면 → 결핍 목록에서 제외 (실제로 아는 것)

> **핵심**: "확신 영역"이 클수록 `[assumed-known]` 위험이 높다.
> 소크라테스의 무지의 지혜: 아는 것이 많다고 느끼는 것이 가장 위험하다.

## Phase 2: 욕망 발동 (Eros Activation)

> "에로스는 아름다운 것 안에서 낳고자 하는 욕망이다" — 향연 206e

### 결핍 → 욕망 변환

결핍 목록에서 AskUserQuestion으로 우선순위를 정한다:

```json
{
  "questions": [{
    "question": "발견된 결핍 중 가장 먼저 해결하고 싶은 것은?",
    "header": "욕망 선택",
    "options": [
      {"label": "{결핍 1}", "description": "{설명}"},
      {"label": "{결핍 2}", "description": "{설명}"},
      {"label": "{결핍 3}", "description": "{설명}"},
      {"label": "모두 동시에", "description": "전체를 관통하는 근본 원인을 찾고 싶다"}
    ],
    "multiSelect": false
  }]
}
```

### 욕망의 구체화

선택된 결핍을 3가지로 변환한다:

1. **탐색 질문**: "왜 X인가?" / "어떻게 X를 Y로 바꿀 수 있는가?"
2. **목표 상태 (아름다운 것)**: 결핍이 해소된 이상적 상태 서술
3. **탐색 범위**: Phase 3에서 어디를 찾아볼지 (코드베이스, 외부 자료, 전문가 의견)

```markdown
## 욕망 정의

**탐색 질문**: {구체적 질문}
**목표 상태**: {결핍이 해소된 이상적 상태}
**탐색 범위**:
- 내부: {코드베이스 경로, 메모리 파일}
- 외부: {검색 키워드, 문서 URL}
```

### Phase 2 완료 기준

- [ ] 결핍 목록에서 최소 1개 우선 결핍 선택됨
- [ ] 탐색 질문이 구체적 질문 형태로 변환됨
- [ ] 목표 상태가 측정 가능한 형태로 정의됨
- [ ] 탐색 범위가 내부/외부로 분류됨

→ 모두 충족 시 Phase 3 (다이몬 중재)로 진행
