## S-01: Landing Page (랜딩 페이지)

### 테스트 목적
- 첫 방문자가 서비스의 핵심 가치를 이해하고 회원가입/로그인으로 전환할 수 있는지 확인
- 모집 중인 앱 미리보기를 통해 서비스 활동성을 인지할 수 있는지 검증

### 사전 준비사항
- 브라우저에서 http://localhost:3000 접속 가능
- 로그아웃 상태 (브라우저 쿠키 삭제 또는 시크릿 모드)
- 데이터베이스에 `status=RECRUITING` 상태인 앱이 최소 1개 이상 존재

### 테스트 절차

#### 1단계: 페이지 접속
1. 브라우저에서 `http://localhost:3000` 접속
2. 페이지 로딩 상태 확인
3. 콘솔 에러 여부 확인 (F12 → Console 탭)

**예상 결과**:
- 페이지가 2초 이내 로딩됨
- 콘솔에 에러 없음
- 히어로 섹션이 화면 상단에 표시됨

#### 2단계: 헤더 요소 확인
1. 페이지 최상단 헤더 영역 확인
2. "TestBridge" 로고 확인
3. 로고 우측에 "개발자로 시작" / "테스터로 시작" 버튼 확인

**예상 결과**:
- 헤더가 고정(sticky) 상태로 스크롤 시에도 상단에 유지됨
- 로고가 명확하게 표시됨
- CTA 버튼이 눈에 띄는 색상으로 강조됨

#### 3단계: 히어로 섹션 확인
1. 메인 헤드라인 문구 확인 (예: "14일 만에 Google Play 출시")
2. 서브 헤드라인 확인 (서비스 핵심 가치 설명)
3. 개발자/테스터 시작 버튼 확인

**예상 결과**:
- 헤드라인이 서비스의 핵심 가치를 명확하게 전달함
- 버튼이 시각적으로 구분됨 (개발자용/테스터용)
- 배경 이미지 또는 그라디언트가 적절하게 표시됨

#### 4단계: 서비스 플로우 시각화 확인
1. 히어로 섹션 아래로 스크롤
2. "어떻게 작동하나요?" 또는 이용 플로우 섹션 확인
3. 단계별 설명 카드 확인 (예: 1단계, 2단계, 3단계)

**예상 결과**:
- 개발자/테스터 각각의 플로우가 명확하게 구분되어 표시됨
- 각 단계에 아이콘 또는 일러스트가 포함됨
- 텍스트가 읽기 쉽게 배치됨

#### 5단계: 샘플 앱 목록 확인
1. 페이지 하단으로 스크롤
2. "모집 중인 앱" 또는 "지금 참여 가능" 섹션 확인
3. 앱 카드가 최대 6개 표시되는지 확인

**예상 결과**:
- 앱 카드가 그리드 레이아웃으로 표시됨 (데스크탑: 3열, 모바일: 1열)
- 각 카드에 다음 정보가 포함됨:
  - 앱 아이콘
  - 앱 이름
  - 카테고리
  - 리워드 금액
  - 남은 자리 또는 마감일

#### 6단계: CTA 버튼 클릭
1. "개발자로 시작하기" 버튼 클릭
2. 페이지 이동 확인
3. 뒤로가기 후 "테스터로 시작하기" 버튼 클릭
4. 페이지 이동 확인

**예상 결과**:
- "개발자로 시작하기" 클릭 시 `/auth/signup?role=developer` 또는 유사 경로로 이동
- "테스터로 시작하기" 클릭 시 `/auth/signup?role=tester` 또는 유사 경로로 이동
- 페이지 전환이 자연스럽게 이루어짐

#### 7단계: 반응형 테스트
1. 브라우저 창 크기를 모바일 크기로 조정 (375px)
2. 레이아웃 변경 확인
3. 모든 요소가 정상적으로 표시되는지 확인

**예상 결과**:
- 헤더가 모바일 레이아웃으로 변경됨 (햄버거 메뉴 또는 세로 정렬)
- 텍스트 크기가 읽기 적절하게 조정됨
- 버튼이 터치하기 적절한 크기로 표시됨
- 가로 스크롤이 발생하지 않음

### 체크리스트

- [ ] 페이지 로딩 시간 2초 이내
- [ ] 헤더에 "TestBridge" 로고 표시됨
- [ ] 히어로 섹션에 메인 헤드라인 표시됨
- [ ] 개발자/테스터 CTA 버튼이 명확하게 구분됨
- [ ] 서비스 이용 플로우가 시각적으로 표현됨
- [ ] 모집 중인 앱 최대 6개 표시됨
- [ ] 앱 카드에 필수 정보(이름, 리워드, 남은 자리) 포함됨
- [ ] CTA 버튼 클릭 시 회원가입 페이지로 이동
- [ ] 모바일 화면에서 레이아웃이 적절하게 변경됨
- [ ] 콘솔 에러 없음

### 주의사항
- 데이터베이스에 `status=RECRUITING` 앱이 없으면 샘플 앱 섹션이 비어있을 수 있습니다
- API 호출(`GET /api/apps?status=RECRUITING&limit=6`)이 실패하면 "모집 중인 앱" 섹션에 에러 메시지 또는 빈 상태가 표시됩니다
- 로그인 상태에서 접속하면 자동으로 대시보드로 리다이렉트될 수 있습니다

## S-02: Signup (회원가입)

### 테스트 목적
- Google OAuth를 통한 소셜 로그인이 정상적으로 작동하는지 확인
- 역할 선택 및 기본 프로필 입력 플로우가 직관적인지 검증
- 이용약관 동의 및 회원 생성이 완료되는지 확인

### 사전 준비사항
- Google 계정 준비 (테스트용 계정 권장)
- 로그아웃 상태
- NextAuth.js 설정 완료 (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` 환경 변수)
- 데이터베이스 연결 정상

### 테스트 절차

#### 1단계: 회원가입 페이지 접속
1. 랜딩 페이지에서 "개발자로 시작하기" 또는 "테스터로 시작하기" 클릭
2. 또는 브라우저에서 직접 `http://localhost:3000/auth/signup` 접속
3. 페이지 로딩 확인

**예상 결과**:
- 회원가입 페이지가 로딩됨
- "Google로 시작하기" 버튼 표시됨
- 이미 로그인 상태이면 대시보드로 자동 리다이렉트됨

#### 2단계: Google OAuth 로그인 시작
1. "Google로 시작하기" 버튼 클릭
2. Google 로그인 팝업 창 또는 새 탭 열림 확인
3. 계정 선택 화면 확인

**예상 결과**:
- Google OAuth 동의 화면이 표시됨
- 팝업 차단기에 의해 차단되지 않음
- 브라우저 주소창에 `accounts.google.com` 도메인 확인

#### 3단계: Google 계정 선택 및 동의
1. 테스트용 Google 계정 선택
2. TestBridge 앱의 권한 요청 확인 (이메일, 프로필 정보)
3. "허용" 또는 "동의" 버튼 클릭

**예상 결과**:
- 권한 요청 내용이 명확하게 표시됨 (이메일 주소, 기본 프로필 정보)
- 동의 버튼 클릭 시 페이지가 닫히고 원래 창으로 돌아옴

#### 4단계: 역할 선택
1. OAuth 인증 완료 후 역할 선택 화면 표시 확인
2. "개발자" / "테스터" / "둘 다" 옵션 확인
3. 원하는 역할 선택 (예: "테스터" 선택)

**예상 결과**:
- 3가지 역할 옵션이 명확하게 구분되어 표시됨
- 선택한 역할이 시각적으로 강조됨 (예: 배경색 변경)
- 각 역할에 대한 간단한 설명이 포함됨

#### 5단계: 기본 프로필 입력
1. 닉네임 입력 필드 확인
2. 닉네임 입력 (예: "테스터홍길동")
3. 프로필 이미지 업로드 버튼 확인 (선택 사항)
4. 기본 이미지 또는 구글 프로필 이미지 확인

**예상 결과**:
- 닉네임 입력 필드가 활성화됨
- 닉네임 유효성 검증 (예: 최소 2자, 최대 20자)
- 프로필 이미지는 선택 사항으로 표시됨
- Google 프로필 이미지가 기본값으로 자동 입력됨

#### 6단계: 이용약관 동의
1. 이용약관 체크박스 확인
2. 이용약관 및 개인정보 처리방침 링크 클릭 (새 탭 열림)
3. 체크박스 클릭하여 동의

**예상 결과**:
- 이용약관 및 개인정보 처리방침 링크가 작동함
- 필수 동의 항목이 명확하게 표시됨
- 체크하지 않으면 다음 단계로 진행 불가

#### 7단계: 회원가입 완료
1. "가입 완료" 또는 "시작하기" 버튼 클릭
2. 로딩 인디케이터 확인
3. API 호출 완료 대기 (`POST /api/users`)
4. 페이지 자동 리다이렉트 확인

**예상 결과**:
- 버튼 클릭 시 로딩 상태 표시됨
- 회원가입 성공 시 역할에 따라 자동 리다이렉트:
  - 개발자 → `/developer`
  - 테스터 → `/tester`
- 에러 발생 시 명확한 에러 메시지 표시 (예: "이미 가입된 계정입니다")

#### 8단계: 회원 생성 확인
1. 대시보드 페이지가 로딩되는지 확인
2. 헤더에 프로필 아이콘 또는 닉네임 표시 확인
3. 데이터베이스에 사용자 레코드 생성 확인 (선택 사항)

**예상 결과**:
- 선택한 역할에 맞는 대시보드가 표시됨
- 헤더에 로그인 상태 정보가 표시됨
- 세션이 정상적으로 유지됨

#### 9단계: 중복 가입 방지 테스트
1. 로그아웃
2. 동일한 Google 계정으로 다시 회원가입 시도
3. 에러 처리 확인

**예상 결과**:
- "이미 가입된 계정입니다" 에러 메시지 표시됨
- 또는 자동으로 로그인 처리되어 대시보드로 이동
- 중복 레코드가 생성되지 않음

### 체크리스트

- [ ] 회원가입 페이지가 정상 로딩됨
- [ ] "Google로 시작하기" 버튼 클릭 시 OAuth 팝업 열림
- [ ] Google 계정 선택 및 권한 동의 완료
- [ ] 역할 선택 화면이 표시됨 (개발자/테스터/둘 다)
- [ ] 닉네임 입력 필드가 작동함
- [ ] 닉네임 유효성 검증 작동 (최소/최대 길이)
- [ ] 이용약관 체크박스 필수 동의 확인
- [ ] 가입 완료 시 역할별 대시보드로 리다이렉트
- [ ] 헤더에 로그인 정보 표시됨
- [ ] 중복 가입 방지 로직 작동

### 주의사항
- **환경 변수 미설정 시** OAuth가 작동하지 않습니다. `.env.local` 파일에 `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` 확인 필요
- **팝업 차단기** 설정이 활성화되어 있으면 Google OAuth 팝업이 차단될 수 있습니다
- **개발 환경** 에서는 Google OAuth Redirect URI에 `http://localhost:3000/api/auth/callback/google` 등록 필요
- **닉네임 중복 검증** 이 구현되지 않았을 경우 동일 닉네임이 허용될 수 있습니다

## S-03: Login (로그인)

### 테스트 목적
- Google OAuth를 통한 로그인이 정상적으로 작동하는지 확인
- 역할별로 올바른 대시보드로 자동 리다이렉트되는지 검증
- 세션 유지 및 인증 상태가 정상적으로 관리되는지 확인

### 사전 준비사항
- 이미 회원가입이 완료된 Google 계정 (S-02 테스트 완료)
- 로그아웃 상태
- NextAuth.js 세션 설정 완료

### 테스트 절차

#### 1단계: 로그인 페이지 접속
1. 브라우저에서 `http://localhost:3000/auth/login` 접속
2. 또는 랜딩 페이지 헤더에서 "로그인" 버튼 클릭
3. 페이지 로딩 확인

**예상 결과**:
- 로그인 페이지가 로딩됨
- "Google로 로그인" 버튼 표시됨
- 회원가입 링크 표시됨 (예: "계정이 없으신가요? 가입하기")

#### 2단계: Google OAuth 로그인 시작
1. "Google로 로그인" 버튼 클릭
2. Google 로그인 팝업 또는 새 탭 열림
3. 계정 선택 화면 확인

**예상 결과**:
- Google OAuth 동의 화면이 표시됨
- 이전에 사용한 계정 목록이 표시됨
- 새 계정으로 로그인 옵션 제공됨

#### 3단계: Google 계정 선택
1. 회원가입 시 사용한 Google 계정 선택
2. 비밀번호 입력 (필요 시)
3. 로그인 완료 대기

**예상 결과**:
- 계정 선택 시 자동으로 로그인 처리됨
- 이미 인증된 계정의 경우 비밀번호 입력 생략됨
- 팝업 창이 닫히고 원래 창으로 돌아옴

#### 4단계: 사용자 정보 조회 및 역할 확인
1. 로그인 완료 후 API 호출 확인 (`GET /api/users/me`)
2. 사용자 역할 확인 (DEVELOPER / TESTER / BOTH)
3. 로딩 인디케이터 확인

**예상 결과**:
- NextAuth.js 세션에 사용자 정보 저장됨
- 사용자 역할이 정확하게 조회됨
- 로딩 상태가 명확하게 표시됨

#### 5단계: 역할별 대시보드 리다이렉트

  - `APPLICATION_APPROVED` (지원 승인)

1. 헤더 알림 아이콘 (종 모양) 클릭

3. 페이지 로딩 확인

**예상 결과**:
- 알림 센터 페이지가 로딩됨

2. 각 알림에 다음 정보 포함 확인:
   - 알림 타입 아이콘
   - 알림 제목 및 내용
   - 발생 시간 (예: "5분 전", "2시간 전", "2일 전")

**예상 결과**:
- 알림이 카드 또는 리스트 형태로 표시됨
- 읽지 않은 알림은 시각적으로 강조됨 (예: 배경색 다름)

- 필터 버튼이 활성화됨
- 읽지 않은 알림만 표시됨
- 읽은 알림은 목록에서 사라짐

3. 모든 알림이 다시 표시되는지 확인

**예상 결과**:
- 필터가 해제됨
- 읽음/안읽음 상관없이 모든 알림 표시됨

3. 알림 상태가 "읽음"으로 변경되는지 확인
4. 해당 페이지로 자동 리다이렉트 확인

**예상 결과**:
- 알림 클릭 시 즉시 읽음 상태로 변경됨
- 알림 타입에 따라 올바른 페이지로 이동:
  - `APPLICATION_APPROVED` → `/tester/apps/[appId]` (앱 상세)
  - `FEEDBACK_RECEIVED` → `/developer/apps/[appId]` (피드백 탭)
  - `REWARD_PAID` → `/tester/participations` (내 테스트 현황)
- 헤더 알림 뱃지 숫자가 1 감소함

#### 6단계: 알림 타입별 라우팅 테스트
1. 여러 알림 타입 클릭
2. 각 알림이 올바른 페이지로 라우팅되는지 확인

**예상 결과**:
- `APPLICATION_APPROVED`: 승인된 앱 상세 페이지
- `FEEDBACK_RECEIVED`: 개발자 앱 상세 > 피드백 탭
- `REWARD_PAID`: 테스터 리워드 내역 또는 참여 현황
- `APP_STATUS_CHANGED`: 앱 상태 변경 페이지
- 기타 타입: 적절한 페이지로 이동

#### 7단계: 전체 읽음 처리
1. "모두 읽음" 또는 "전체 읽음 처리" 버튼 확인
2. 버튼 클릭
3. 확인 모달 표시 (선택 사항)
4. API 호출 확인 (`PATCH /api/notifications/mark-all-read`)
5. 모든 알림이 읽음 상태로 변경되는지 확인

**예상 결과**:
- 전체 읽음 버튼이 명확하게 표시됨
- 버튼 클릭 시 모든 알림이 읽음 상태로 변경됨
- 알림 목록의 시각적 표시가 모두 변경됨 (배경색 등)
- 헤더 알림 뱃지가 0으로 변경됨

#### 8단계: 페이지네이션 테스트
1. 알림 목록 하단으로 스크롤
2. "다음 페이지" 또는 페이지 번호 클릭
3. API 호출 확인 (`GET /api/notifications?page=2&limit=20`)
4. 다음 페이지 알림 표시 확인

**예상 결과**:
- 페이지네이션 버튼이 표시됨 (20개 초과 시)
- 다음 페이지 클릭 시 새로운 알림 로딩됨
- URL이 `?page=2`로 변경됨
- 이전 페이지로 돌아가기 가능

#### 9단계: 빈 알림 목록 테스트
1. 모든 알림을 읽음 처리
2. "읽지 않음" 필터 적용
3. 빈 상태 화면 확인

**예상 결과**:
- "새 알림이 없습니다" 또는 유사 메시지 표시됨
- 빈 상태 일러스트 또는 아이콘 표시됨
- 필터를 "전체"로 변경하면 읽은 알림이 표시됨

#### 10단계: 헤더 알림 뱃지 확인
1. 알림 센터에서 뒤로가기
2. 헤더 알림 아이콘 확인
3. 읽지 않은 알림 개수 뱃지 확인
4. 알림 아이콘 클릭 시 드롭다운 또는 센터 페이지 이동 확인

**예상 결과**:
- 헤더 알림 아이콘에 빨간색 뱃지 표시됨 (숫자)
- 뱃지 숫자가 실제 읽지 않은 알림 개수와 일치함
- 알림 아이콘 클릭 시 알림 센터로 이동

#### 11단계: 실시간 알림 업데이트 테스트 (선택 사항)
1. 다른 계정 또는 백엔드에서 새 알림 생성
2. 알림 센터 페이지 새로고침 없이 알림 추가 확인
3. 헤더 뱃지 자동 업데이트 확인

**예상 결과**:
- 실시간 알림 업데이트 구현 시: 새로고침 없이 알림 추가됨
- WebSocket 또는 SSE 미구현 시: 새로고침 필요

### 체크리스트

- [ ] 알림 센터 페이지가 정상 로딩됨
- [ ] 알림 목록이 시간 역순으로 표시됨
- [ ] 읽지 않은 알림이 시각적으로 강조됨
- [ ] "읽지 않음" 필터 작동
- [ ] "전체" 필터 작동
- [ ] 알림 클릭 시 읽음 처리됨
- [ ] 알림 타입별 올바른 페이지로 라우팅됨
- [ ] "모두 읽음" 버튼 클릭 시 전체 읽음 처리
- [ ] 페이지네이션 작동 (20개 초과 시)
- [ ] 빈 알림 목록 시 적절한 메시지 표시
- [ ] 헤더 알림 뱃지 숫자가 정확함
- [ ] 콘솔 에러 없음

### 주의사항
- **알림 데이터 부족** 시 페이지네이션 및 필터링 테스트가 제한될 수 있습니다
- **실시간 알림 업데이트** 는 WebSocket 또는 SSE 구현이 필요하며, 미구현 시 새로고침 필요
- **알림 타입별 라우팅** 은 알림 테이블의 `metadata` 필드에 저장된 정보를 기반으로 합니다
- **헤더 알림 뱃지** 는 API 호출(`GET /api/notifications?isRead=false&limit=100`) 또는 WebSocket으로 업데이트됩니다

## 공통 주의사항

### 환경 설정
- **데이터베이스**: PostgreSQL 연결 확인 (`.env` 파일의 `DATABASE_URL`)
- **Google OAuth**: `.env.local` 파일에 `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` 설정 필수
- **NextAuth.js**: `NEXTAUTH_URL`, `NEXTAUTH_SECRET` 설정 필수
- **포트**: 프론트엔드 `http://localhost:3000`, 백엔드 API는 동일 포트 (Next.js API Routes)

### 브라우저 호환성
- Chrome 최신 버전 권장
- Safari, Firefox 호환 테스트 권장
- 모바일 브라우저 (iOS Safari, Android Chrome) 테스트 권장

### 테스트 데이터 준비
- 회원가입 시 사용할 Google 계정 준비
- 데이터베이스에 샘플 앱, 알림 데이터 시드(seed) 추가 권장
- Prisma Studio 활용: `npx prisma studio`

### 에러 처리
- API 에러 발생 시 명확한 에러 메시지 표시 확인
- 네트워크 오류 시 재시도 옵션 확인
- 폼 검증 에러 시 필드별 에러 메시지 표시 확인

### 성능
- 페이지 로딩 시간 2초 이내 권장
- API 응답 시간 500ms 이내 권장
- 이미지 최적화 확인 (WebP, lazy loading)
