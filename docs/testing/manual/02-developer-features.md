## 사전 준비사항

### 필수 조건
- DEVELOPER 또는 BOTH 역할의 계정으로 로그인 필요
- Google OAuth 인증 완료
- 크레딧 잔액 확인 (앱 등록 시 필요 시)

### 테스트 계정 생성 방법
1. `/auth/signup` 접속
2. Google 계정으로 로그인
3. 역할 선택 시 "개발자" 또는 "둘 다" 선택
4. 닉네임 입력 후 가입 완료
5. `/developer` 경로로 자동 리다이렉트 확인

## D-01: 개발자 대시보드

### 📋 테스트 목적
개발자의 전체 테스트 현황을 한눈에 파악할 수 있는지 확인

### 📍 접근 경로
- URL: `/developer`
- 네비게이션: 로그인 후 자동 리다이렉트 또는 사이드바 "대시보드" 클릭

---

### 🧪 테스트 절차

#### 1단계: 대시보드 진입
**동작**:
1. DEVELOPER 역할 계정으로 로그인
2. `/developer` 경로 접속

**예상 결과**:
- 페이지 타이틀: "개발자 대시보드" 또는 "Dashboard" 표시
- 헤더에 사용자 프로필 이미지 및 이름 표시
- 사이드바에 "대시보드", "내 앱", "앱 등록" 메뉴 표시

---

#### 2단계: 주요 지표 확인
**동작**:
1. 대시보드 상단 요약 카드 확인
2. 각 카드의 숫자 표시 확인

**예상 결과**:
- **진행 중인 앱 수**: `IN_TESTING` 상태 앱 개수 표시
- **총 참여자 수**: 승인된 참여자 총합
- **평균 별점**: 모든 앱의 평균 별점 (1~5점)
- **피드백 수**: 제출된 피드백 총 개수
- 숫자가 실시간으로 정확하게 표시됨

---

#### 3단계: 진행 중인 테스트 확인
**동작**:
1. "진행 중인 테스트" 섹션 확인
2. 각 앱 카드 내용 확인

**예상 결과**:
- `IN_TESTING` 상태 앱들이 카드 형태로 나열
- 각 카드에 다음 정보 표시:
  - 앱 이름 및 아이콘
  - 진행률 프로그레스 바 (승인된 인원 / 목표 인원)
  - D-Day 카운트다운 (예: D-7, D-Day, +3일 경과)
  - 평균 별점 (별 아이콘)
  - 피드백 수
- 진행 중인 앱이 없는 경우 "진행 중인 테스트가 없습니다" 메시지 표시

---

#### 4단계: 최근 피드백 확인
**동작**:
1. "최근 피드백" 섹션 확인
2. 최대 3개의 피드백 항목 확인

**예상 결과**:
- 최신 피드백 3개가 시간 역순으로 표시
- 각 항목에 다음 정보 포함:
  - 앱 이름
  - 테스터 닉네임
  - 별점 (1~5점)
  - 코멘트 미리보기 (첫 50자)
  - 작성 시간 (예: 2시간 전, 1일 전)
- "더 보기" 링크 → 앱 상세 페이지의 피드백 탭으로 이동

---

#### 5단계: Quick Actions 확인
**동작**:
1. 대시보드 우측 또는 하단 Quick Actions 영역 확인
2. 각 버튼 클릭

**예상 결과**:
- **"앱 등록" 버튼**: `/developer/apps/new` 이동
- **"지원자 관리" 버튼**: `/developer/apps` 이동 또는 최근 앱 상세로 이동
- 버튼 클릭 시 페이지 전환 정상 동작

---

#### 6단계: 구독 플랜 정보 확인
**동작**:
1. 구독 플랜 카드 또는 배너 확인
2. 크레딧 잔액 표시 확인

**예상 결과**:
- 현재 구독 플랜 표시 (예: Free, Pro, Enterprise)
- 크레딧 잔액 표시 (예: 50,000 크레딧)
- 플랜 업그레이드 버튼 (선택 사항)

---

### ✅ 체크리스트

- [ ] 대시보드 페이지 로딩 완료 (3초 이내)
- [ ] 주요 지표 4개 카드 모두 표시 (진행 중인 앱 수, 총 참여자 수, 평균 별점, 피드백 수)
- [ ] 진행 중인 테스트 카드에 D-Day 정확히 계산됨
- [ ] 진행률 프로그레스 바가 실제 인원에 맞게 표시
- [ ] 최근 피드백 3개 표시 (또는 "피드백이 없습니다" 메시지)
- [ ] Quick Actions 버튼들이 정상 동작
- [ ] 크레딧 잔액 정확히 표시
- [ ] 모바일 화면에서도 레이아웃 깨지지 않음
- [ ] 네비게이션 메뉴 정상 동작

---

### ⚠️ 주의사항

- **D-Day 계산**: `테스트 시작일 + 14일 - 현재 날짜`
  - 예: 3월 1일 시작 → 3월 7일: D-7
  - 예: 3월 1일 시작 → 3월 15일: D-Day
  - 예: 3월 1일 시작 → 3월 17일: +2일 경과
- **평균 별점**: 소수점 1자리까지 표시 (예: 4.5점)
- **진행률**: 승인된 참여자 / 목표 인원 (예: 12/20명 = 60%)
- **API 에러 처리**: 네트워크 에러 시 "데이터를 불러올 수 없습니다" 메시지 표시

## D-02: 앱 등록

### 📋 테스트 목적
개발자가 4단계 위저드를 통해 앱을 정상적으로 등록할 수 있는지 확인

### 📍 접근 경로
- URL: `/developer/apps/new`
- 네비게이션: 사이드바 "앱 등록" 클릭 또는 대시보드 "앱 등록" 버튼

---

### 🧪 테스트 절차

#### 1단계: 기본 정보 입력
**동작**:
1. `/developer/apps/new` 접속
2. 위저드 1단계 "기본 정보" 확인
3. 다음 필드 입력:
   - 앱 이름: "테스트 앱 A"
   - 패키지명: "com.example.testa"
   - 카테고리: 드롭다운에서 선택 (예: "게임")
   - 앱 설명: 최소 50자 이상 입력
4. 앱 아이콘 이미지 업로드 (선택 사항)
5. "다음" 버튼 클릭

**예상 결과**:
- 위저드 상단에 "1/4 단계" 표시
- 모든 필수 필드에 빨간색 별표(*) 표시
- 패키지명 형식 검증:
  - 올바른 형식: `com.example.app` (통과)
  - 잘못된 형식: `invalid-package` (에러 메시지)
- 앱 설명 최소 글자 수 검증 (50자 미만 시 에러)
- 필수 필드 누락 시 "다음" 버튼 비활성화 또는 에러 메시지
- 입력 완료 시 2단계로 자동 전환

---

#### 2단계: 테스트 설정
**동작**:
1. 위저드 2단계 "테스트 설정" 확인
2. 다음 필드 입력:
   - 목표 인원: 20 (슬라이더 또는 입력)
   - 테스트 기간: 14일 (기본값)
   - Google Play 비공개 테스트 링크: `https://play.google.com/apps/test/com.example.testa`
3. "다음" 버튼 클릭

**예상 결과**:
- 위저드 상단에 "2/4 단계" 표시
- 목표 인원 검증:
  - 최소: 14명 (Google Play 요구사항)
  - 최대: 100명
  - 범위 벗어나면 에러 메시지
- 테스트 기간 검증:
  - 최소: 14일 (기본값, 수정 불가)
- Google Play 링크 형식 검증:
  - 올바른 형식: `https://play.google.com/...` (통과)
  - 잘못된 형식: 에러 메시지
- 입력 완료 시 3단계로 전환

---

#### 3단계: 리워드 설정
**동작**:
1. 위저드 3단계 "리워드 설정" 확인
2. 다음 필드 입력:
   - 리워드 타입: "포인트" 선택 (또는 "상품권", "기타")
   - 리워드 금액: 5000 입력 (원 단위)
3. "다음" 버튼 클릭

**예상 결과**:
- 위저드 상단에 "3/4 단계" 표시
- 리워드 금액 검증:
  - 최소: 1,000원
  - 최대: 제한 없음 (또는 크레딧 잔액 내)
  - 1,000원 미만 시 에러 메시지
- 총 비용 표시: `목표 인원 × 리워드 금액` (예: 20명 × 5,000원 = 100,000원)
- 크레딧 잔액 부족 시 경고 메시지 (선택 사항)
- 입력 완료 시 4단계로 전환

---

#### 4단계: 피드백 설정 및 제출
**동작**:
1. 위저드 4단계 "피드백 설정" 확인
2. 피드백 항목 선택:
   - [x] UI/UX
   - [x] 성능 (Performance)
   - [x] 기능성 (Functionality)
   - [x] 안정성 (Stability)
   - [ ] 기타 (선택 사항)
3. 앱 스크린샷 업로드 (최대 5장)
4. "미리보기" 버튼 클릭하여 입력 내용 확인
5. "제출" 버튼 클릭

**예상 결과**:
- 위저드 상단에 "4/4 단계" 표시
- 피드백 항목은 최소 1개 이상 선택 필수
- 스크린샷 업로드:
  - 이미지 형식 검증 (JPG, PNG만 허용)
  - 파일 크기 제한 (예: 5MB 이하)
  - 최대 5장까지 업로드
  - 업로드된 이미지 썸네일 미리보기 표시
- "미리보기" 클릭 시 모달 또는 새 창에서 앱 상세 미리보기
- "제출" 클릭 시:
  - 로딩 스피너 표시
  - API 호출: `POST /api/apps`
  - 성공 시 `/developer/apps` 또는 앱 상세 페이지로 리다이렉트
  - 실패 시 에러 메시지 표시 (예: "패키지명 중복", "네트워크 에러")

---

#### 5단계: 제출 후 확인
**동작**:
1. 앱 등록 완료 후 리다이렉트된 페이지 확인
2. 내 앱 목록에서 방금 등록한 앱 확인

**예상 결과**:
- 성공 토스트 메시지: "앱이 성공적으로 등록되었습니다"
- 내 앱 목록(`/developer/apps`)에 새 앱 표시
- 앱 상태: `PENDING_APPROVAL` (검토 중) 표시
- 앱 카드에 등록한 정보 정확히 반영:
  - 앱 이름, 아이콘
  - 상태 뱃지: "검토 중"
  - 목표 인원 표시

---

#### 6단계: 위저드 중간 이탈 테스트
**동작**:
1. 위저드 2단계에서 "뒤로" 버튼 클릭
2. 1단계로 돌아가는지 확인
3. 입력한 데이터 유지 여부 확인
4. 위저드 도중 페이지 닫고 다시 접속

**예상 결과**:
- "뒤로" 버튼 클릭 시 이전 단계로 이동
- 이전 단계에서 입력한 데이터 유지됨 (폼 상태 보존)
- 위저드 도중 페이지 닫으면 데이터 손실 (또는 로컬 스토리지에 임시 저장, 선택 사항)
- 페이지 재접속 시 1단계부터 다시 시작

---

### ✅ 체크리스트

- [ ] 위저드 4단계 모두 정상 표시 (1/4 → 2/4 → 3/4 → 4/4)
- [ ] 패키지명 형식 검증 정상 동작 (com.example.app 형식만 허용)
- [ ] 목표 인원 범위 검증 (14~100명)
- [ ] 리워드 금액 최소값 검증 (1,000원 이상)
- [ ] 총 비용 계산 정확 (목표 인원 × 리워드 금액)
- [ ] 스크린샷 업로드 정상 동작 (최대 5장, 이미지 형식 검증)
- [ ] 피드백 항목 최소 1개 선택 필수
- [ ] "미리보기" 기능 정상 동작
- [ ] "뒤로" 버튼으로 이전 단계 이동 시 데이터 유지
- [ ] 제출 성공 시 내 앱 목록에 새 앱 표시
- [ ] 제출 실패 시 에러 메시지 표시 (패키지명 중복, 네트워크 에러 등)
- [ ] 모바일 화면에서도 위저드 정상 동작

---

### ⚠️ 주의사항

- **패키지명 중복 검증**: 이미 등록된 패키지명은 에러 메시지 표시
  - 예: "이미 등록된 패키지명입니다"
- **크레딧 잔액**: 총 비용이 크레딧 잔액을 초과하면 경고 또는 차단 (선택 사항)
- **Google Play 링크**: 실제 유효한 링크인지 검증하지 않음 (형식만 검증)
- **스크린샷**: 필수 아님 (0장도 허용), 최대 5장까지만
- **위저드 진행 상태**: 브라우저 새로고침 시 데이터 손실 (로컬 스토리지 미사용 시)
- **API 에러 처리**:
  - 400: 검증 실패 (필드별 에러 메시지 표시)
  - 409: 패키지명 중복
  - 500: 서버 에러 (일반 에러 메시지)

## D-03: 내 앱 목록

### 📋 테스트 목적
개발자가 등록한 모든 앱을 상태별로 조회하고 관리할 수 있는지 확인

### 📍 접근 경로
- URL: `/developer/apps`
- 네비게이션: 사이드바 "내 앱" 클릭 또는 헤더 메뉴

---

### 🧪 테스트 절차

#### 1단계: 앱 목록 조회
**동작**:
1. `/developer/apps` 접속
2. 페이지 로딩 확인

**예상 결과**:
- 페이지 타이틀: "내 앱" 또는 "My Apps" 표시
- 등록한 앱들이 그리드 레이아웃으로 나열 (2~3열)
- 앱이 없는 경우:
  - "등록된 앱이 없습니다" 메시지
  - "앱 등록하기" CTA 버튼 → `/developer/apps/new` 이동
- 로딩 중: 스켈레톤 UI 또는 로딩 스피너 표시

---

#### 2단계: 상태별 필터링
**동작**:
1. 페이지 상단의 필터 탭 확인
2. 각 상태 탭 클릭:
   - "전체"
   - "검토 중" (PENDING_APPROVAL)
   - "모집 중" (RECRUITING)
   - "테스트 중" (IN_TESTING)
   - "완료" (COMPLETED)
   - "프로덕션" (IN_PRODUCTION)

**예상 결과**:
- 각 탭에 해당 상태의 앱 개수 표시 (예: "모집 중 (3)")
- 탭 클릭 시 필터링된 앱 목록만 표시
- 해당 상태의 앱이 없으면 "해당 상태의 앱이 없습니다" 메시지
- URL 쿼리 파라미터 변경 (예: `/developer/apps?status=RECRUITING`)

---

#### 3단계: 앱 카드 정보 확인
**동작**:
1. 각 앱 카드의 표시 정보 확인
2. 여러 상태의 앱 카드 비교

**예상 결과**:
각 카드에 다음 정보 표시:
- **앱 아이콘**: 업로드한 이미지 또는 기본 아이콘
- **앱 이름**: 최대 30자 (넘으면 말줄임표)
- **상태 뱃지**:
  - `PENDING_APPROVAL`: 노란색 "검토 중"
  - `RECRUITING`: 파란색 "모집 중"
  - `IN_TESTING`: 초록색 "테스트 중"
  - `COMPLETED`: 회색 "완료"
  - `IN_PRODUCTION`: 보라색 "프로덕션"
- **진행률 프로그레스 바**: 승인된 인원 / 목표 인원 (예: 12/20)
  - 퍼센트 표시 (60%)
  - 색상: 50% 미만 빨간색, 50~80% 주황색, 80% 이상 초록색
- **D-Day**: 테스트 종료까지 남은 일수 (예: D-7, D-Day, +3일 경과)
  - `IN_TESTING` 상태만 표시
- **평균 별점**: 별 아이콘 + 숫자 (예: ⭐ 4.5)
  - 피드백이 있는 경우만 표시

---

#### 4단계: 앱 카드 클릭
**동작**:
1. 앱 카드 클릭
2. 앱 상세 페이지 이동 확인

**예상 결과**:
- 앱 카드 클릭 시 `/developer/apps/[id]` 이동
- 해당 앱의 상세 페이지 정상 로딩
- 브라우저 뒤로 가기 시 목록 페이지로 복귀 (스크롤 위치 유지, 선택 사항)

---

#### 5단계: 정렬 및 검색
**동작**:
1. 정렬 드롭다운 확인 (있는 경우)
2. 검색창에 앱 이름 입력 (있는 경우)

**예상 결과**:
- **정렬 옵션** (선택 사항):
  - "최신순" (기본값)
  - "이름순"
  - "D-Day 임박순"
- **검색** (선택 사항):
  - 앱 이름으로 실시간 필터링
  - Debounce 적용 (300ms 지연)
  - 검색 결과 없으면 "검색 결과가 없습니다" 메시지

---

#### 6단계: 페이지네이션
**동작**:
1. 앱이 10개 이상인 경우 페이지네이션 확인
2. 다음 페이지 버튼 클릭

**예상 결과**:
- 페이지당 12개 앱 표시 (또는 설정값)
- 페이지네이션 버튼: 이전, 1, 2, 3, ..., 다음
- 현재 페이지 강조 표시
- URL 쿼리 파라미터 변경 (예: `/developer/apps?page=2`)
- 페이지 이동 시 스크롤 최상단으로 이동

---

### ✅ 체크리스트

- [ ] 앱 목록 페이지 로딩 완료 (3초 이내)
- [ ] 그리드 레이아웃 정상 표시 (2~3열)
- [ ] 상태별 필터 탭 정상 동작 (6개 상태)
- [ ] 각 앱 카드에 필수 정보 표시 (아이콘, 이름, 상태, 진행률)
- [ ] 진행률 프로그레스 바 정확히 계산됨
- [ ] D-Day 정확히 계산됨 (`IN_TESTING` 상태만)
- [ ] 평균 별점 정확히 표시 (피드백 있는 경우만)
- [ ] 앱 카드 클릭 시 상세 페이지 이동
- [ ] 앱이 없는 경우 "등록된 앱이 없습니다" 메시지 + CTA 버튼
- [ ] 모바일 화면에서 1열 레이아웃으로 변경
- [ ] 페이지네이션 정상 동작 (10개 이상 앱 시)

---

### ⚠️ 주의사항

- **진행률 계산**: `(승인된 참여자 수 / 목표 인원) × 100`
  - 예: 12명 승인 / 20명 목표 = 60%
- **D-Day 표시**:
  - `IN_TESTING` 상태만 표시
  - 다른 상태는 D-Day 미표시 또는 "모집 중", "완료" 등의 텍스트
- **상태 뱃지 색상**:
  - 검토 중: 노란색 (yellow)
  - 모집 중: 파란색 (blue)
  - 테스트 중: 초록색 (green)
  - 완료: 회색 (gray)
  - 프로덕션: 보라색 (purple)
- **평균 별점**: 피드백이 0개인 경우 "평가 없음" 또는 미표시
- **API 에러 처리**: 네트워크 에러 시 "앱 목록을 불러올 수 없습니다" 메시지

## D-04: 앱 상세 / 테스트 관리

### 📋 테스트 목적
개발자가 앱의 상세 정보를 확인하고, 지원자를 관리하며, 피드백을 수집할 수 있는지 확인

### 📍 접근 경로
- URL: `/developer/apps/[id]`
- 네비게이션: 내 앱 목록에서 앱 카드 클릭

---

### 🧪 테스트 절차

#### 1단계: 앱 상세 페이지 진입
**동작**:
1. 내 앱 목록에서 특정 앱 카드 클릭
2. `/developer/apps/[id]` 접속

**예상 결과**:
- 페이지 타이틀: 앱 이름 표시
- 헤더 영역에 앱 기본 정보 표시:
  - 앱 아이콘
  - 앱 이름
  - 패키지명
  - 상태 뱃지
  - 카테고리
  - 평균 별점
- 5개 탭 메뉴 표시:
  1. 현황 (Overview)
  2. 지원자 (Applications)
  3. 참여자 (Participants)
  4. 피드백 (Feedbacks)
  5. 가이드 (Guide)

---

#### 2단계: 탭 1 - 현황 (Overview)
**동작**:
1. "현황" 탭 클릭 (기본 활성화)
2. 실시간 통계 확인

**예상 결과**:
- **주요 지표 카드** (4개):
  - **총 지원자**: `PENDING` 상태 지원서 개수
  - **승인된 인원**: `APPROVED` 상태 지원서 개수
  - **평균 별점**: 모든 피드백의 평균 (1~5점)
  - **피드백 수**: 제출된 피드백 총 개수
- **진행률 프로그레스 바**:
  - 승인된 인원 / 목표 인원 (예: 12/20 = 60%)
  - 시각적 프로그레스 바
- **D-Day 카운트다운**:
  - `IN_TESTING` 상태인 경우만 표시
  - 예: "테스트 종료까지 7일 남음"
- **앱 설명**: 등록 시 입력한 앱 설명 전체 표시
- **스크린샷 갤러리**: 업로드한 스크린샷 Carousel 또는 그리드 표시

---

#### 3단계: 탭 2 - 지원자 (Applications)
**동작**:
1. "지원자" 탭 클릭
2. 대기 중인 지원자 목록 확인
3. 지원자 프로필 확인
4. 승인/거절 버튼 클릭

**예상 결과**:
- **지원자 목록**:
  - `PENDING` 상태 지원서만 표시
  - 각 항목에 다음 정보:
    - 테스터 프로필 이미지
    - 닉네임
    - 신뢰도 점수 (예: 85/100)
    - 테스트 완료율 (예: 12/15 = 80%)
    - 자기소개 미리보기 (첫 50자)
    - 기기 정보 (예: Galaxy S24 / Android 14)
    - 지원 시간 (예: 2시간 전)
- **지원자 프로필 확인**:
  - 닉네임 클릭 시 모달 또는 확장 영역에서 상세 프로필 표시
  - 과거 테스트 이력, 신뢰도 점수 계산 근거
- **승인 버튼**:
  - 클릭 시 확인 모달: "이 지원자를 승인하시겠습니까?"
  - 승인 시 API 호출: `PATCH /api/applications/[id]` (status: APPROVED)
  - 성공 시:
    - 해당 지원서가 "참여자" 탭으로 이동
    - 토스트 메시지: "지원자가 승인되었습니다"
    - 승인 인원 = 목표 인원이면 앱 상태 자동 변경: `RECRUITING` → `IN_TESTING`
- **거절 버튼**:
  - 클릭 시 확인 모달: "이 지원자를 거절하시겠습니까?"
  - 거절 시 API 호출: `PATCH /api/applications/[id]` (status: REJECTED)
  - 성공 시:
    - 해당 지원서가 목록에서 사라짐
    - 토스트 메시지: "지원자가 거절되었습니다"
- **일괄 승인 기능** (선택 사항):
  - 체크박스로 여러 지원자 선택
  - "선택 항목 일괄 승인" 버튼 클릭
  - 선택한 지원자 모두 승인

**주의사항**:
- 승인 인원 = 목표 인원이면 더 이상 승인 불가 (버튼 비활성화 또는 에러 메시지)
- 거절된 지원서는 복구 불가 (또는 관리자 권한 필요)

---

#### 4단계: 탭 3 - 참여자 (Participants)
**동작**:
1. "참여자" 탭 클릭
2. 승인된 참여자 목록 확인
3. 각 참여자의 옵트인 상태 확인

**예상 결과**:
- **참여자 목록**:
  - `APPROVED` 상태 지원서 + participation 레코드가 있는 사용자
  - 각 항목에 다음 정보:
    - 테스터 프로필 이미지
    - 닉네임
    - 참여 상태:
      - `ACTIVE`: 활성 (초록색)
      - `DROPPED_OUT`: 이탈 (빨간색)
      - `FEEDBACK_SUBMITTED`: 피드백 제출 완료 (파란색)
      - `REWARD_PAID`: 리워드 지급 완료 (보라색)
    - 옵트인 상태: "옵트인 완료" / "옵트인 대기 중"
    - 가입일 (참여 시작일)
    - 경과 일수 (예: 7일째)
- **이탈 알림**:
  - 옵트인하지 않은 채로 3일 경과 시 경고 표시
  - 빨간색 배지: "이탈 위험"
- **참여자 상세**:
  - 닉네임 클릭 시 참여 상세 정보 모달
  - 제출한 피드백 링크 (피드백 제출 완료 시)

---

#### 5단계: 탭 4 - 피드백 (Feedbacks)
**동작**:
1. "피드백" 탭 클릭
2. 제출된 피드백 목록 확인
3. 각 피드백 클릭하여 상세 확인

**예상 결과**:
- **피드백 목록**:
  - 제출된 모든 피드백 표시 (시간 역순)
  - 각 항목에 다음 정보:
    - 테스터 닉네임 + 프로필 이미지
    - 전체 별점 (1~5점, 별 아이콘)
    - 코멘트 미리보기 (첫 100자)
    - 제출 시간 (예: 2일 전)
    - 버그 리포트 여부 (버그 아이콘 표시)
- **피드백 상세 모달**:
  - 피드백 클릭 시 모달 또는 확장 영역에서 전체 내용 표시:
    - 전체 별점
    - **항목별 별점**:
      - UI/UX: 4점
      - 성능: 5점
      - 기능성: 4점
      - 안정성: 3점
    - 전체 코멘트 (줄바꿈 유지)
    - **버그 리포트** (있는 경우):
      - 버그 제목
      - 상세 설명
      - 기기 정보
      - 스크린샷 (클릭 시 확대)
- **피드백 통계** (상단 요약):
  - 평균 별점: 4.2/5.0
  - **항목별 평균**:
    - UI/UX: 4.0
    - 성능: 4.5
    - 기능성: 4.2
    - 안정성: 4.0
  - 총 피드백 수: 12개
  - 버그 리포트 수: 3개

---

#### 6단계: 탭 5 - 가이드 (Guide)
**동작**:
1. "가이드" 탭 클릭
2. Google Play Console 설정 가이드 확인

**예상 결과**:
- **가이드 내용**:
  - Google Play Console 비공개 테스트 설정 단계별 가이드
  - 스크린샷 또는 다이어그램 포함
  - 주요 단계:
    1. Google Play Console 접속
    2. 앱 선택 → 테스트 → 비공개 테스트
    3. 트랙 생성 (예: "내부 테스트")
    4. 테스터 목록 추가 (이메일 입력)
    5. 버전 업로드 및 검토 요청
  - **테스터 이메일 목록 복사 기능**:
    - "참여자" 탭의 승인된 테스터 이메일 자동 생성
    - "이메일 복사" 버튼 클릭 시 클립보드에 복사
    - 예: `tester1@gmail.com, tester2@gmail.com, ...`
- **추가 리소스**:
  - Google Play Console 공식 문서 링크
  - TestBridge 고객 지원 링크

---

#### 7단계: 프로덕션 확인 버튼
**동작**:
1. 앱이 `COMPLETED` 상태인 경우 (14일 완료)
2. "프로덕션 확인" 버튼 확인 (페이지 우측 상단)
3. 버튼 클릭

**예상 결과**:
- **버튼 표시 조건**:
  - 앱 상태: `COMPLETED`
  - 테스트 기간 14일 완료
  - 피드백 수 >= 목표 인원의 50% (예: 20명 목표 시 최소 10개 피드백)
- **버튼 클릭 시**:
  - 확인 모달: "이 앱을 프로덕션으로 확인하시겠습니까?"
  - 확인 시 API 호출: `PATCH /api/apps/[id]` (status: IN_PRODUCTION)
  - 성공 시:
    - 앱 상태 변경: `COMPLETED` → `IN_PRODUCTION`
    - 토스트 메시지: "프로덕션으로 확인되었습니다"
    - 상태 뱃지 업데이트
- **버튼 비활성화 조건**:
  - 앱 상태가 `COMPLETED`가 아님
  - 피드백 수가 부족함

---

### ✅ 체크리스트

- [ ] 앱 상세 페이지 로딩 완료 (3초 이내)
- [ ] 헤더에 앱 기본 정보 표시 (아이콘, 이름, 패키지명, 상태)
- [ ] 5개 탭 모두 정상 표시 (현황, 지원자, 참여자, 피드백, 가이드)
- [ ] **현황 탭**: 주요 지표 4개 카드 표시, D-Day 정확히 계산
- [ ] **지원자 탭**: 대기 중인 지원자 목록 표시, 승인/거절 버튼 정상 동작
- [ ] 승인 시 참여자 탭으로 이동 확인
- [ ] 승인 인원 = 목표 인원 시 앱 상태 자동 변경 (`RECRUITING` → `IN_TESTING`)
- [ ] **참여자 탭**: 승인된 참여자 목록 표시, 옵트인 상태 확인
- [ ] 이탈 위험 알림 표시 (옵트인 미완료 + 3일 경과)
- [ ] **피드백 탭**: 제출된 피드백 목록 표시, 항목별 별점 확인
- [ ] 피드백 클릭 시 상세 모달 정상 동작
- [ ] 버그 리포트 있는 경우 스크린샷 확대 기능
- [ ] **가이드 탭**: Google Play Console 설정 가이드 표시
- [ ] 테스터 이메일 목록 복사 기능 정상 동작
- [ ] 프로덕션 확인 버튼 표시 조건 정확 (COMPLETED 상태 + 피드백 충분)
- [ ] 모바일 화면에서도 탭 네비게이션 정상 동작

### ⚠️ 주의사항

- **승인/거절 권한**: 앱 소유자(개발자)만 가능
  - 다른 사용자가 URL로 직접 접근 시 403 에러 또는 버튼 비활성화
- **승인 인원 제한**: 승인 인원 = 목표 인원이면 더 이상 승인 불가
  - 추가 승인 시도 시 에러 메시지: "목표 인원에 도달했습니다"
- **앱 상태 자동 전환**:
  - 승인 인원 = 목표 인원 → `RECRUITING` → `IN_TESTING`
  - 14일 경과 + 피드백 수집 → `IN_TESTING` → `COMPLETED` (자동 또는 수동)
- **D-Day 계산**:
  - `IN_TESTING` 상태일 때만 표시
  - 계산식: `테스트 시작일 + 14일 - 현재 날짜`
- **피드백 통계**:
  - 평균 별점: 모든 피드백의 평균 (소수점 1자리)
  - 항목별 평균: 각 항목의 별점 평균 (소수점 1자리)
- **API 에러 처리**:
  - 404: 앱을 찾을 수 없음
  - 403: 권한 없음 (다른 개발자의 앱)
  - 500: 서버 에러
